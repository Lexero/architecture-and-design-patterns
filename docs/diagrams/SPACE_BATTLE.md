# Архитектура микросервисов игры Космический бой

## Диаграммы

- [services.puml](services.puml) - все микросервисы, БД, брокер, клиентские приложения
- [sequence_battle.puml](sequence_battle.puml) - диаграмма последовательностей для одного боя
- [sequence_tournament.puml](sequence_tournament.puml) - диаграмма последовательностей для турнира

---

## Набор микросервисов

| Сервис                   | Ответственность                                   |
|--------------------------|---------------------------------------------------|
| **User Service**         | Регистрация, вход, выдача и валидация JWT         |
| **Tournament Service**   | Турниры, заявки, расписание, результаты           |
| **Game Server**          | Игровая логика                                    |
| **Rating Service**       | Рейтинг игроков и турниров                        |
| **Replay Service**       | Запись кадров боя, воспроизведение для Player App |
| **Notification Service** | Push / Email уведомления                          |

Клиентские приложения:

- **Web App** - интерфейс для управления профилем, турнирами, рейтингом.
- **Player App** - просмотр текущего и прошедшего боя.
- **Агент** - загружает программу управления кораблями и отправляет команды на Game Server.

Инфраструктура:

- **API Gateway** - единая точка входа.
- **Message Broker (Kafka)** - асинхронный обмен событиями между сервисами.

## Узкие места и масштабирование

### 1. Game Server - обработка событий в реальном времени

**Проблема:** Каждая активная игра держит WebSocket-соединения. При тысячах одновременных боёв один инстанс не справится.

**Решения:**

- Горизонтальное масштабирование со **Sticky Sessions** по `gameId`.
- Кеширование состояния игры в **Redis**
- Распределение маршрутизации через API Gateway - новые игры равномерно распределяются по подам.

### 2. Message Broker - узкое горлышко при нагрузке

**Проблема:** Все сервисы читают/пишут через брокер; при пиковых нагрузках (большой турнир) очереди могут накапливаться.

**Решения:**

- Партиционирование топиков по `gameId` - параллельная обработка событий разных игр.
- Группы консьюмеров для разных сервисов

### 3. Rating Service - гонка при параллельных обновлениях

**Проблема:** После одновременных боёв рейтинг одного игрока может обновляться конкурентно.

**Решения:**

- Партиционирование очереди по `userId` - события для одного игрока обрабатываются строго последовательно.
- **Optimistic locking** с версионированием строк в PostgreSQL.

### 4. Replay Service - большой объём данных

**Проблема:** 60 fps × длительность боя × N одновременных игр = гигабайты данных.

**Решения:**

- Удаление старых реплаев
- Компрессия реплаев

## Компоненты с высокой частотой изменений (OCP)

### 1. Game Server - команды и игровая логика

**Вид изменений:** Новые типы команд, изменение физики, балансировка.

**Решение:** IoC-контейнер с динамической регистрацией - новая команда добавляется как новый класс + регистрация в IoC
без изменения ядра.

### 2. Tournament Service - форматы турниров и правила участия

**Вид изменений:**
- Разные форматы турниров
- Рейтинговые пороги участия
- Правила начисления очков

**Решение:** Strategy pattern - загородиться интерфейсом `TournamentFormatInterface`  и вызывать разные стратегии в зависимости от турнира

### 3. Rating Service - формула рейтинга

**Вид изменений:**
- Переход с простой формулы на ELO, Glicko-2
- Учёт рейтинга турнира
- Сезонные изменения

**Решение:** Strategy pattern - загородиться интерфейсом `RatingCalculatorInterface` и вызывать разные стратегии в зависимости от игры

### 4. Notification Service - каналы доставки

**Вид изменений:**
- Добавление новых каналов нотификаций: Telegram, Discord, мобильных push
- Разные языки уведомлений
- Персонализация уведомлений

**Решение:** `NotificationChannelInterface`
